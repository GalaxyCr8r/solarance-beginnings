Regression Diagnosis · 2026-05-27 · Server-side

Asteroids stopped spawning.

The asteroid-population timer still exists and still runs — but the M4 server refactor (#90) quietly changed its firing interval from 60 seconds to 12 hours. Because each tick spawns at most one asteroid per sector, fields that used to fill in minutes now take days to weeks — and on a fresh publish the first asteroid doesn't appear for half a day. Functionally: no asteroids.

Suspect commit: 5d17348 (M4 refactor, #90) Primary file: server/src/lifecycle/timers.rs:28 Status: diagnosed, fix proposed

§How asteroid spawning is supposed to work

Everything is server-side, exactly as expected. There is no init-time asteroid seeding — the upkeep timer is the only thing in the codebase that creates asteroids. That makes its cadence the single point of failure.

init()
Seeds two asteroid_sector rows (Alpha, Beta). No asteroids created here.
sector_upkeep timer
Scheduled reducer. Should fire every 60s; now fires every 12h.
asteroid_sector_upkeep()
For one sector: if asteroid count < sparseness×10, spawn one.
create_asteroid()
Creates the stellar object + asteroid row at a random in-field position.

§Root cause

The upkeep interval was changed from 60s to 12h

Primary

The refactor (5d17348) collapsed a two-tier timer system into one and, in the process, set the surviving timer's interval to 12 hours while leaving the "Every Minute" comment in place. 

server/src/lifecycle/timers.rs · initialize()- // pre-refactor: server/src/types/sectors/timers.rs
-     dsl.create_sector_upkeep_timer(
-         ScheduleAt::Interval(Duration::from_secs(60).into())   // 60 seconds
-     )?;

+ // current: server/src/lifecycle/timers.rs:27
+     dsl.create_sector_upkeep_timer(CreateSectorUpkeepTimer {
+         scheduled_at: ScheduleAt::Interval(Duration::from_hours(12).into()), // Every Minute  <-- stale comment
+     })?;
⬢ Why this reads as a complete outage, not just "slow"

ScheduleAt::Interval fires the first time only after one full interval elapses. On a freshly published / cleared database, that means zero asteroids for the first 12 hours, then one per sector, then nothing for another 12 hours. No realistic play or test session ever sees an asteroid appear — indistinguishable from "spawning is broken."

The arithmetic

The spawn helper creates exactly one asteroid per call when a sector is under target — it does not backfill to the target. So time-to-populate is (target count) × (tick interval).

SectorTarget@ 60s (intended)@ 12h (current)
Alpha10~10 min~5 days
Beta50~50 min~25 days
Plus a one-interval cold start: +60s intended vs +12h now before the very first asteroid.

§Compounding issues found nearby

Not the root cause, but they made the regression invisible and could independently kill spawning. Worth fixing while you're in here.

The helper's error is silently swallowed

Compounding 
server/src/logic/sectors/mod.rs:33-35 · sector_upkeep()for sector in dsl.get_all_asteroid_sectors() {
    let _ = asteroid_sector_upkeep(ctx, &dsl, §or.get_id());  // Result discarded
}

The let _ = throws away the Result. If a sector's upkeep fails — bad sector lookup, a rejected server check, a create_asteroid error — nothing reaches spacetime logs. This directly violates the Debugging Contract in CLAUDE.md ("helpers propagate errors, they don't swallow them"). It's also why this regression had to be found by reading code rather than reading logs.

try_server_only() is brittle and runs twice per tick

Latent risk

sector_upkeep calls try_server_only(&dsl)?, then the helper it calls runs the same check again. The check matches ctx.sender() against three hard-coded identity substrings (server/src/utility.rs:14-29). For a scheduled reducer the sender is the module's own identity — so if the "big server-side refactor in prep for this new MCP" republished the module under an identity that isn't in that allowlist, the top-level ? aborts every tick before a single asteroid is considered. The refactor commit message itself flags this: "try_server_only() is quite brittle but I just want to get this off the ground."

⬢ How to rule it out in 30 seconds

Run spacetime logs <db> and grep for "Denied server request from:". If it's firing on every sector_upkeep tick, the deployed module identity isn't in the allowlist and that's a second, independent outage hiding behind the swallowed error above. If it's absent, the interval is the whole story.

§The fix

1. Restore the interval (required)

server/src/lifecycle/timers.rs:27-29  dsl.create_sector_upkeep_timer(CreateSectorUpkeepTimer {
-     scheduled_at: ScheduleAt::Interval(Duration::from_hours(12).into()), // Every Minute
+     scheduled_at: ScheduleAt::Interval(Duration::from_secs(60).into()), // Every minute
  })?;

This matches both the stale comment and the reducer's own doc-comment ("Runs every 60 seconds", logic/sectors/mod.rs:26). Republish the module — init only runs on first publish, so an existing database keeps its old 12h timer row until cleared or migrated.

2. Surface the swallowed error (recommended)

server/src/logic/sectors/mod.rs:33-35  for sector in dsl.get_all_asteroid_sectors() {
-     let _ = asteroid_sector_upkeep(ctx, &dsl, §or.get_id());
+     if let Err(e) = asteroid_sector_upkeep(ctx, &dsl, §or.get_id()) {
+         log::warn!("asteroid upkeep failed for sector {:?}: {}", sector.get_id(), e);
+     }
  }

3. Sanity-check try_server_only against the deployed identity (verify)

  • Confirm the published module's identity matches one of the allowlisted substrings in utility.rs, or the timer is dead on arrival regardless of interval.
  • Longer term: replace the hard-coded substring list with the actual scheduled-reducer sender check; the brittleness is acknowledged in-tree.
⬢ Verification after the fix

Publish with --clear-database to a scratch DB, wait ~60 seconds, then query the asteroid table. You should see Alpha begin to fill (one new asteroid per minute up to 10) and Beta up to 50. spacetime logs should be quiet — no "Denied server request" lines.
